# Bug Record: BUG-2026-05-11-001

## Summary

- Title: Android overlay does not appear or react from Settings
- Current status: `fix-attempted`
- Severity: `high`
- Owner: codex
- Next command: `/sf-ship BUG-2026-05-11-001`

## Reproduction

- Preconditions:
  - Android real device.
  - Installed debug APK reports `build: sha=7d6fd2d | run=25662735886 | ref=master`.
  - Overlay permission is granted.
  - Accessibility permission is granted.
- Steps:
  1. Open WinGlows.
  2. Go to Settings.
  3. Use the overlay enable/start control.
  4. Observe whether the native floating overlay appears or any Settings status changes.

## Expected Behavior

- The Settings overlay control should call the native overlay bridge.
- With overlay permission granted, `requestedEnabled` should become true and the foreground overlay service should start.
- A native floating overlay bubble should appear.
- If Android blocks the foreground service or `WindowManager.addView`, Settings diagnostics should show a concise native failure code.

## Observed Behavior

- The overlay does not appear.
- The user reports that the overlay button does not trigger anything.
- Diagnostic still reports `enabled=false`, `requested=false`, and `running=false` even though `overlay_permission=true` and `accessibility_permission=true`.
- Existing `recent_events` contains tab and screen-build activity, but no overlay action event, making the current logs insufficient to identify whether the failure is UI tap handling, Dart bridge invocation, native preference persistence, service startup, or overlay view attachment.
- On 2026-05-19 with APK sha=37116dd run=26091472372, the user reports starting overlay blocks the whole interface; tapping it stops recording but no useful result follows.
- The latest diagnostic now shows the Flutter/native call path is reached (`overlay_start`, `overlay_start_result`, `last_native_event=bridgeCall method=startOverlayRecording`) but the overlay still reports `running=false` and `service_state=created`.
