# Bug Record: BUG-2026-05-24-003

## Summary

- Title: Action row scroll preview starts from the wrong page
- Current status: `fixed-pending-verify`
- Severity: `medium`
- Owner: Codex
- Next command: Android IME physical-device retest

## Reproduction

- Preconditions:
  - Android IME action bar has a horizontally paged scroll row with at least two pages.
- Steps:
  1. Scroll the action row to the second page.
  2. Start another horizontal scroll gesture from that second page.
  3. Observe the shrinking/transition preview during the drag.

## Expected Behavior

- The scroll transition starts from the page currently visible before the drag.
- The previewed keys preserve spatial continuity, so the user can understand where they are in the action row.

## Observed Behavior

- The transition appears to start from the first page even when the user begins dragging from the second page.
- The visible keys during the animation do not match the keys that were in front of the user just before the gesture.

## Evidence

- Operator report in chat, 2026-05-24.
- Code path reviewed in `WinGlowzKeyboardView.handleHorizontalRowScroll` and `drawScrollableRow`.

## Diagnosis Notes

| Date UTC | Skill | Author | Hypothesis | Findings | Outcome |
|----------|-------|--------|------------|----------|---------|
| 2026-05-24 18:23:51 UTC | sf-fix | Codex | The paged-row drag start is anchored to a stale/intermediate scroll offset instead of the persisted current row page. | `handleHorizontalRowScroll` set `horizontalGestureStartOffset` from `horizontalRowScrollOffsetById`; if a slow snap/return animation left that offset near page 0 while `rowPageById` represented page 1, the preview could restart from page 0. | supported |
