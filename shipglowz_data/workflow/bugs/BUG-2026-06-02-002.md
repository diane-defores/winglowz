# Bug Record: BUG-2026-06-02-002

## Summary

- Title: Long-press swipe target keys show letters and dispatch primary text
- Current status: `fixed-pending-verify`
- Severity: `medium`
- Owner: Diane
- Next command: `/sf-test --retest BUG-2026-06-02-002`

## Reproduction

- Preconditions:
  - WinGlowz Android IME is active.
  - Corner mode is enabled with the Smart French/punctuation corner preset.
- Steps:
  1. Start the long-press swipe mode from an eligible action key.
  2. Observe highlighted letter keys that have directional corner actions.
  3. Move to `C`, which has an `Up` corner assignment for `8`.
  4. Release on `C`.

## Expected Behavior

- Highlighted target keys show only the action that will be performed.
- `C` displays and dispatches `8`, not the primary letter `c`.
- Text keys without an eligible swipe action do not insert their primary letter during this mode.
- Multi-action targets resolve one action by the configured priority order, then show and dispatch only that action.

## Observed Behavior

- Some highlighted middle letter keys still showed their alphabetic label.
- Releasing on `C` did not input `8`.
- The UI still showed corner glyph clutter on target keys instead of a single active action label.

## Evidence

- User report in chat on 2026-06-02.
- `KeyboardCornerShortcutsTest` already validates `letter-c` has `cornerAssignments.up?.label == "8"`.
- `WinGlowzKeyboardView.tryDispatchAfterLongPressSwipe` dispatched the target key with `GestureSelection.PrimaryTap`.
- `WinGlowzKeyboardView.drawKey` rendered `displayLabelForRect(key, drawRect)` even when the key was a long-press swipe target.
