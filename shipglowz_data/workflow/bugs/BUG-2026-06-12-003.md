# Bug Record: BUG-2026-06-12-003

## Summary

- Title: Keyboard mic permission denial leaves the user without guided recovery
- Current status: `fix-attempted`
- Severity: `medium`
- Owner: Diane
- Next command: `/107-sf-test`

## Reproduction

- Preconditions:
  - WinGlows keyboard is active on Android.
  - Microphone permission for WinGlows is denied.
- Steps:
  1. Open a text field with the WinGlows keyboard.
  2. Tap the microphone action.
  3. Observe the action row and status message.

## Expected Behavior

- The keyboard should show a clear blocked state when microphone permission is missing.
- The status should be visually distinct as an error.
- The user should have an immediate recovery path toward the microphone onboarding/settings flow.

## Observed Behavior

- The keyboard only showed `Microphone permission required`.
- The action button did not clearly transform into a blocked/help state.
- No recovery affordance was exposed from the same interaction.

## Evidence

- Operator report in chat on 2026-06-12.
- `KeyboardVoiceController.start()` already detects missing `RECORD_AUDIO`, but only emits a plain text state.
- The IME view had no dedicated blocked mic visual state and no follow-up action on the mic button.
