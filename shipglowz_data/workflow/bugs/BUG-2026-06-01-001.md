# Bug Record: BUG-2026-06-01-001

## Summary

- Title: Keyboard theme presets stop following light/dark mode after non-color options
- Current status: `fixed-pending-verify`
- Severity: `medium`
- Owner: Diane
- Next command: `/sf-fix BUG-2026-06-01-001`

## Reproduction

- Preconditions:
  - Android keyboard theme preset is selected from Settings or the native IME theme panel.
  - App appearance is changed through `Système`, `Clair`, or `Sombre`.
- Steps:
  1. Select a keyboard preset such as Glass Mint or Pixel Candy.
  2. Change a non-color keyboard option, for example `Relief clavier`.
  3. Toggle app appearance from light to dark, dark to light, or system auto.

## Expected Behavior

- Keyboard presets keep their selected identity and automatically render the light or dark palette matching the global app/system appearance.
- Non-color options such as relief, opacity, spacing, and press effects remain intact.

## Observed Behavior

- The automatic light/dark switch is tied to exact equality with the stock preset config.
- Once a non-color option changes the stored config, equality fails and the keyboard can keep the previous palette instead of switching variants.

## Evidence

- `WinGlowzKeyboardView.resolvedThemeConfigForMode` only switches variants when the whole config equals the light or dark preset.
- `SettingsScreen._setKeyboardRelief` stores a changed theme config, making exact equality fail for otherwise preset-controlled themes.
